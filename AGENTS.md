# AGENTS.md

이 문서는 Codex가 **항상 먼저 읽는** 진입점이다. 짧게 유지한다. 세부 내용은 각 참조 파일에 있다.

> 지침 파일은 `docs/`에 있다. **이 문서의 모든 파일 경로는 레포 루트 기준**이다. (AGENTS.md만 루트에 둔다.)

---

## 1. 이 프로젝트에서 너(Codex)의 역할

- 너는 **구현 담당**이다. **설계자가 아니다.**
- 시스템의 구조와 설계는 이미 `docs/architecture/` 폴더의 문서에 정의되어 있다. 너는 그 스펙을 **그대로** 구현한다.
- 스펙에 없는 구조적 결정을 **임의로 만들어내지 않는다.** (예: 새 클래스 계층, 새 통신 방식, 새 매니저 도입 등)
- 만들려는 것이 이 프로젝트에 **이미 있는 것과 같은 종류의 일**(예: 복제 프로퍼티 등록 방식, GameplayTag 등록, AttributeSet 정의 패턴, 시스템별 architecture 문서 구조)이면, 새 방식을 발명하지 말고 **기존 관례를 먼저 찾아 그대로 따른다.** 관례가 있는지 불확실하면 2항대로 멈추고 묻는다.

## 2. 가장 중요한 행동 규칙

**불확실하면 멈추고 질문한다. 추측해서 진행하지 않는다.**

- 스펙이 모호하거나, 두 가지 해석이 가능하거나, `docs/architecture/` 문서에 답이 없으면 → **구현을 멈추고 무엇이 불명확한지 묻는다.**
- "아마 이런 의도겠지" 하고 질러서 코드를 만들지 않는다. 잘못된 추측으로 만든 코드는 되돌리는 비용이 추측을 아끼는 이득보다 크다.
- 단, 네이밍/포맷/사소한 구현 디테일처럼 `docs/conventions/`로 답할 수 있는 건 묻지 말고 규칙대로 처리한다.

## 3. 작업 순서

1. 관련 `docs/architecture/[시스템].md`를 읽는다.
2. `docs/constraints.md`의 금지 패턴에 위배되지 않는지 확인한다.
3. `docs/conventions/coding.md`에 맞춰 구현한다.
4. 작업 단위가 끝나면 `docs/conventions/commit.md` 규칙대로 커밋한다.

## 4. 문서-코드 동기화 (중요)

- 구현하면서 `docs/architecture/` 문서의 내용과 **실제로 달라진 부분이 생기면**, 임의로 코드만 바꾸지 말고 **그 사실을 먼저 보고한다.**
- 문서가 거짓이 되는 순간 이 프로젝트 전체가 무너진다. 코드와 문서 중 무엇을 진실로 삼을지는 **사람이 판단**한다.

## 5. 참조 파일 지도

| 알고 싶은 것 | 읽을 파일 |
|---|---|
| 이 프로젝트가 무엇이고 무엇을 증명하는가 | `docs/project.md` |
| 언리얼/GAS에서 절대 하면 안 되는 것 | `docs/constraints.md` |
| 코드 네이밍·구조 규칙 | `docs/conventions/coding.md` |
| 커밋 메시지 형식 | `docs/conventions/commit.md` |
| 특정 시스템을 어떻게 구현하는가 | `docs/architecture/[시스템].md` |
| 마찰/스펙 모호를 어디에 적나 | `docs/friction-log.md` |

## 6. 보고 원칙

- 모든 응답의 **첫 줄에 현재 날짜·시간**을 적는다. 포맷: `연/월/일(요일) - 시:분:초` (24시간). **추측하지 말고 실제 시각을 쓴다** — 셸에서 `date +"%Y/%m/%d(%a) - %H:%M:%S"`로 확인.
- 매 프롬프트마다 장황한 작업 보고서를 쓰지 않는다.
- 기록은 **커밋 메시지**로 남긴다. 별도 산문 보고는 사람이 요청할 때만 작성한다.
- 단, **2항(불확실해서 멈추고 질문한 경우)** 또는 **4항(문서-코드 불일치를 발견한 경우)**이 발생하면, 보고와 함께 `docs/friction-log.md`에 한 줄을 남긴다. (형식은 그 파일 안에 명시되어 있다.)
