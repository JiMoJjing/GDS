# 제약 및 금지 패턴

> 이 문서는 GAS 사용법 튜토리얼이 아니다. **Codex(및 사람)가 언리얼/네트워크 작업에서 실제로 자주 틀리는 것**만 모은 방지 목록이다. 여기 적힌 건 "스타일 권장"이 아니라 **위반 시 버그가 나는 규칙**으로 취급한다.

**대상 언리얼 버전: UE 5.4**
(Enhanced Input은 5.4에서 표준이다. 5.5 이상에서 추가/변경된 API를 끌어다 쓰지 않는다 — 5.4에 존재하지 않으면 쓰지 않는다.)

---

## 1. 서버/클라이언트 권위

- 게임플레이에 영향을 주는 판정(데미지, 상태 변경, 점수, 스폰 등)은 **반드시 서버에서** 한다.
- 클라이언트 입력은 **요청**일 뿐이다. 서버가 검증하고 결정한다.
- 권위가 필요한 로직 앞에는 권위 체크를 명시한다. (`HasAuthority()` 확인 누락은 가장 흔한 버그다.)
- 클라이언트에서 직접 Attribute를 깎거나 게임 상태를 바꾸지 않는다.

## 2. GAS

- GameplayEffect는 **서버에서 적용**한다. 예측(prediction)이 필요한 경우에만 정해진 prediction 패턴을 따른다.
- ASC 초기화 타이밍을 지킨다. Pawn/PlayerState 구조에 따라 서버와 클라이언트에서 초기화 지점이 다르다. (`PossessedBy`, `OnRep_PlayerState` 등) — 임의의 위치에서 초기화하지 않는다.
- Attribute 값 변경은 항상 GameplayEffect를 통한다. 직접 setter로 갈기지 않는다. (Meta Attribute / Execution Calculation 패턴 준수.)
- GameplayTag는 문자열 하드코딩하지 않는다. 정해진 태그 등록 방식을 따른다.

## 3. 초기화 타이밍

- `BeginPlay`에서 다른 액터/컴포넌트가 이미 준비됐다고 가정하지 않는다. 네트워크 환경에서는 복제 순서가 보장되지 않는다.
- 복제된 변수에 의존하는 초기화는 `OnRep_` 콜백에서 처리한다. 생성자/`BeginPlay`에서 복제값을 기다리지 않는다.

## 4. 리플리케이션

- 복제할 필요 없는 것을 복제하지 않는다. (불필요한 `Replicated` 프로퍼티는 곧 대역폭 낭비다.)
- 모든 게 모두에게 보여야 하는 건 아니다. 팀 기반 정보는 `ReplicationCondition`으로 대상 클라이언트를 제한한다.
- RPC는 `Server`/`Client`/`Multicast` 의미를 정확히 구분해 쓰고, `Server` RPC에는 검증(validation)을 붙인다.

## 5. API / 환경

- deprecated API를 쓰지 않는다. **UE 5.4 기준의 현행 API**를 사용한다. 5.5 이상에서만 존재하는 API는 쓰지 않는다. 확실치 않으면 멈추고 묻는다.
- 이 프로젝트는 **C++ 우선**이다. Blueprint 전용 동작을 전제로 설계하지 않는다.
- 매직 넘버/하드코딩 경로를 피한다. 설정값은 정해진 위치(Config, DataAsset 등)에 둔다.
