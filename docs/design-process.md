# 설계 세션 절차 (사람 ↔ Claude)

> 이 문서는 한 시스템을 **채팅에서 함께 설계하는** 절차다. Codex는 읽지 않는다 (AGENTS.md 참조 지도에 넣지 않는다).
> retro.md가 인수 검증 *이후*의 지침 보수 절차라면, 이 문서는 구현 *이전*의 설계 절차다.
> 산출물은 `architecture/[system].md` (= `_template.md`를 채운 것). 이 문서는 그 산출물에 **도달하는 길**이지 산출물 자체가 아니다.

---

## 1. 원칙

- **고도(altitude): 결정·계약·경계만 정한다.** 함수 시그니처·변수·함수 본문은 정하지 않는다.
  판단 기준 — "Codex가 틀리면 구조를 다시 짜야 하는가" → 정한다. "유능한 UE 개발자면 알아서 같게 할 디테일인가" → 안 정하고 맡긴다.
- **한 번에 하나씩.** Claude가 진행을 끌고, 아래 체크리스트를 순서대로 한 포인트(또는 묶음)씩 짚는다. 한꺼번에 쏟지 않는다. 사람이 결정·입력하면 다음으로 넘어간다.
- **막히면 선택지로.** 사람이 답을 못 정하면 Claude가 2~4개 갈래로 잘라 장단과 함께 제시한다. 빈칸으로 두고 넘어가지 않는다.
- **결정마다 "왜".** 근거와 **탈락 대안**을 같이 박는다. 이게 `_template.md` 2항이자 포트폴리오 재료다.
- **한 세션 = 한 시스템.** 새 채팅에서 시작한다.

## 2. 세션 흐름

순서대로 짚되, 앞이 정해지면 뒤가 따라 나오므로 기계적으로 전부 묻지 않는다. 특히 5번이 잡히면 6~7은 대부분 거기서 파생된다.

**A. 열기 — 게이트·범위**
1. 무엇을 설계하나 (한 줄)
2. `project.md`의 어떤 증명 목표를 담당하나 — 해당 없으면 왜 만드는지부터 (스코프 게이트)
3. 범위: 들어가는 것 / 의도적으로 빼는 것 — "넣고 싶다"와 "증명에 필요하다" 구분
4. 출력/계약: 끝났을 때 눈에 보이는 결과

**B. 본체 — 기능 설계 (척추 = 상태·전이)**
5. **상태와 전이:** 가질 수 있는 상태, 상태 간 전이, 각 전이의 트리거 — *여기를 먼저 또렷이 그린다.*
6. 규칙/판정: 무엇이 허용되고 무엇이 거절되나, 그 판정 로직
7. 흐름과 분기 + 경계·실패: 정상 흐름, 그리고 잘못된 상태에서 호출·도중 중단·무효 입력을 어떻게 처리하나
8. 관계·의존, 책임 경계: 무엇을 읽고/쓰고/알리나, 공유 상태는 누가 소유하나, 어떤 클래스가 무엇을 맡나 (함수·변수 아님)
9. 권위·가시성: 누가 판정하나(서버), 무엇이 누구에게 보이나(복제), 복제 조건, prediction 유무

**C. 닫기 — 핸드오프**
10. 페이즈 경계: "사람이 눈으로 확인 가능한 동작"이 처음 나오는 지점들 (P0, P1…)
11. 인수 검증 시나리오: 서버 + 클라 PIE로 어떻게 확인하나
12. 미룰 것: 지금 안 정해도 나머지 설계가 안 막히는 것 — "추후"로 명시

## 3. 종료

- 위 내용을 `_template.md` 틀에 채워 `architecture/[system].md`로 정리한다. **빈 칸을 남기지 않는다** (칸이 비면 Codex가 추측한다).
- 이 세션에서 지침 파일을 건드렸으면 `docs/` + 프로젝트 파일 양쪽을 동기화한다.
