# 회고 절차 (지침 업그레이드)

> 이 문서는 Claude ↔ 사람 사이의 *지침 보수* 절차다. Codex는 읽지 않는다 (AGENTS.md 참조 지도에 넣지 않는다).
> 목적: 파이프라인을 돌며 나온 마찰을 **증거에 근거해** 지침에 반영하되, 지침 비대화를 막는다.

## 1. 원칙
- **수집은 상시, 적용은 경계에서 배치.** 흐름 중간에 지침을 고치지 않는다 (churn 방지).
- **증거 없이 고치지 않는다.** 근거는 diff(`git log`) 또는 마찰 로그. "그냥 느낌"은 승격 사유가 아니다.
- **승격 bar: 반복되거나 비용이 큰 것만** 상시 지침으로 올린다. 일회성은 그 자리에서 고치고 잊는다.
- **지침 변경 최종 승인은 사람.** 모든 후속 작업의 동작을 바꾸므로 자동 적용하지 않는다.

## 2. 언제
시스템 하나가 **인수 검증(architecture 8항) 통과**한 시점 = 회고 1회 단위.
(시스템이 길어 마찰이 쌓이면 중간에 1회 더 가능. 단 기본 단위는 시스템 끝.)

## 3. 신호 → 증거 → 패치 대상
| 신호가 나온 단계 | 증거 | 패치 대상 |
|---|---|---|
| 설계 (Claude와 빙빙 돌거나 템플릿이 안 물어봄) | 그 깨달음 | `_template.md` / 프로젝트 지침 |
| Codex 핸드오프 (스펙 모호 → 질문·오해) | Codex 질문·오답 | 해당 `architecture/[system].md`; 반복되면 `constraints.md`/`coding.md` |
| 결과 diff (feat 직후 fix, 같은 파일 반복 수정) | `git log` + diff | `constraints.md`/`coding.md`/`conventions/` |

## 4. 절차
1. **(상시) 마찰 잡기** — 거슬린 것 한 줄씩 `friction-log.md`에 적는다. diff성 증거는 git이 이미 기록하므로 따로 안 적어도 됨.
2. **(경계) 회고 실행** — 마지막 회고 이후 `git log -p` + `friction-log.md`를 Claude에 넘긴다.
3. Claude가 3항 표대로 분류하고, 승격 항목은 **근거(diff/로그)를 인용해** 지침 수정 초안을 만든다.
4. 사람이 승인 → 지침 수정 → `docs/` + 프로젝트 파일 **양쪽 동기화** → `docs(...)` 커밋.
5. `friction-log.md`를 비운다. (durable 기록은 위 커밋과 지침 diff. 로그는 일회용 작업 메모.)

## 5. 승격하지 않는 것 (비대화 방지)
- 일회성 오류 → 그 자리에서 고치고 로그에서 버린다.
- 설계 자체가 틀린 것 → 지침 문제 아님. 재설계로 처리.
