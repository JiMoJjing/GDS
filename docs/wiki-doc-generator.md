# wiki-doc-generator — 설계 문서 → 포폴/Velog 기술문서 변환 가이드

> 이 문서는 Claude ↔ 사람 절차 문서다. **Codex는 읽지 않는다** (AGENTS.md 참조 지도에 넣지 않는다 — `design-process.md`·`retro.md`와 같은 계열).
> 목적: 한 시스템의 `architecture/[system].md`(+ 그 시스템의 git 기록·설계 세션 메모)를 **포트폴리오/Velog에 바로 쓸 수 있는 서술형 기술문서**로 펼친다.
> 트리거: "위키 문서 작성해" / "이 시스템 포폴 글로 정리해" 같은 지시.

---

## 1. 이 가이드의 위치

설계 파이프라인의 **마지막 칸**이다.

```
설계(design-process) → architecture/[system].md → Codex 구현 → 인수 검증 → [여기] 포폴 문서
```

`architecture/[system].md`는 **Codex 핸드오프용 스펙**이라 압축돼 있고 불릿 위주다. 이 가이드는 그걸 **사람(미래의 나 / 포폴 심사자)이 읽는 서술형 문서**로 다시 푼다. 새 설계가 아니라 **이미 확정된 설계의 해설**이다.

## 2. 입력 (없으면 지어내지 않는다)

1. **`architecture/[system].md`** — 주 재료. 특히 2항(설계 의도/핵심 결정)·5항(네트워크)이 글의 척추다.
2. **그 시스템의 git 기록** — `git log` + diff에서 해당 scope(`gas`/`net`/`weapon` 등) 커밋. 커밋 메시지의 "왜"가 글의 근거가 된다.
3. **설계 세션 메모** — 채팅에서 오간 결정·탈락 대안 중 architecture 문서에 다 안 담긴 것.

> 입력이 부실하면(문서가 빈약하거나 git 기록이 얕으면) **추측으로 채우지 말고 사람에게 보강을 요청**한다. 이 프로젝트의 "불확실하면 멈춘다" 정신은 포폴 문서에도 적용된다 — 지어낸 설계 근거는 심사에서 가장 먼저 무너진다.

## 3. 문서 구조 (기본 골격)

`_template.md`의 항목을 **서술형으로 펼치는** 순서다. 시스템 성격에 따라 가감하되, 기본은 이 순서를 따른다 (시스템마다 구조가 같으면 포폴 전체가 일관되게 읽힌다).

1. **무엇을, 왜** — 이 시스템이 `project.md`의 **어떤 증명 목표**를 담당하나, 왜 이 프로젝트에 필요했나. (표면 기능 설명이 아니라 "이게 왜 증명 대상인가".)
2. **설계 결정과 근거** — 핵심 결정 + 왜 그렇게 했나 + **택하지 않은 대안**. (← architecture 2항. 이 섹션이 글의 중심이다.)
3. **구조와 책임 분리** — 핵심 클래스/계층, 소유·관계, 누가 무엇을 책임지나. (← architecture 3항.)
4. **네트워크 설계** — 무엇이 서버 권위인가 / 무엇을 복제하고 안 하나 / 복제 조건(`ReplicationCondition`)과 그 이유 / prediction 유무 / 최적화(Relevancy·Dormancy·Lag Compensation·Pooling). (← architecture 5항. **GDS의 헤드라인 증명이라 별도 섹션으로 강제한다** — 네트워크 항목이 비면 이 프로젝트의 포폴 가치가 안 드러난다.)
5. **데이터·실행 흐름** — 입력부터 결과까지 한 흐름을 서술로. (← architecture 4항.)
6. **설계 패턴·재사용성** — 어떤 책임 분리/패턴으로 수렴했나, 이 패턴이 왜 유효하고 다른 시스템에도 재사용 가능한가. (포폴 재가공 시 가장 중요한 재료 — 밀도 있게.)

## 4. 탈락 대안을 어떻게 다루나 (★ 친구 원본과의 핵심 차이)

친구 원본 프롬프트(`포폴_제작용_기술_백업.md`)는 "폐기안·비교안·시행착오 전부 제외"가 절대 규칙이다. 그건 그쪽이 **망가진 기존 구조를 고친 이야기**라 "최종 구조가 처음부터 정답이었던 것처럼" 쓰는 게 맞기 때문이다.

**GDS는 다르다.** greenfield + 증명형 프로젝트라, **택하지 않은 대안과 그 탈락 사유가 곧 설계 깊이의 증거**다 (`_template.md` 2항이 이걸 포폴 재료로 명시한다). 그래서 GDS에서는 대안을 **지우지 않는다.** 대신 **형태를 구분**한다:

- **유지 (깊이):** "A·B·C를 두고 B를 택했다. A는 X 때문에, C는 Y 때문에 탈락." → **확정된 설계 판단의 트레이드오프**로 서술. 이게 심사자가 보고 싶은 부분이다.
- **제외 (노이즈):** "처음엔 A로 했다가 안 돼서 B로 바꿨다", "헤맸다", "이것저것 시도했다" → **시행착오 회고**. 작업 로그지 설계 문서가 아니다. 뺀다.

판단 기준: **"이게 설계 시점에 내릴 수 있었던 합리적 비교인가(유지) vs 구현 중 헤맨 흔적인가(제외)".**

## 5. 문체 규칙 (친구 원본에서 그대로 가져온 것 — 좋음)

- **서술형 문단 중심.** 불릿 나열로 끝내지 않는다. 보조 불릿·소제목·표는 필요할 때만.
- **"무엇을 했다"가 아니라 "왜 이 구조로 수렴했는가"** 가 드러나게. 작업 로그가 아니라 **설계 기준서**.
- **코드 전문을 길게 붙이지 않는다.** 구현 구조·설계 패턴 중심으로 설명하고, 클래스/함수/시스템명은 설계를 설명하는 데 필요한 범위에서만 인용한다.
- **연결 표현 활용** — "핵심은", "즉", "결국", "이 결정의 이유는" 등으로 인과를 잇되, 장황한 수사는 피한다.
- **한글 서술, 코드·식별자·API는 영어** (GDS 출력 규칙).

## 6. 제외 규칙

- 시행착오 회고("처음엔 ~하려고 했다"), 폐기된 실험 경로, 진행 과정 감상·메타 설명.
- 중복 설명, 불필요한 도입/마무리 멘트.
- **예외:** 4항의 "유지" 대상(확정 설계의 비교 대안)은 제외하지 않는다 — 이게 친구 원본과 갈리는 지점.

## 7. 출력

- **마크다운.** Velog/노션에 바로 붙여넣을 수 있는 완성형.
- 제목은 문서 상단에 1개. 섹션은 `#`/`##` 헤더.
- 길어도 괜찮다 — 각 섹션을 충분히 상세하게. 특히 4항(네트워크 설계)·6항(설계 패턴)은 밀도 있게.
- 파일로 저장할지 채팅 본문으로 뽑을지는 사람이 그때 지정한다.
